CHIFFRES VISIBLES :

ARCHITECTURE :
  ✅ Firestore Rules — deny by default
  ✅ ProtectedRoute → /unauthorized
  ✅ Rôle Admin — accès complet
  ✅ Rôle Sous-admin — pas de suppression
  ✅ Rôle Comptable — paiements/factures
  ✅ Rôle Étudiant — ses données uniquement

┌─ À DIRE AU JURY ─────────────────────────────────────────────────────────────
│
│  "Le contrôle d'accès par rôle, ou RBAC, est implémenté à deux niveaux.
│
│   Côté backend, les Firestore Rules appliquent un principe de deny by
│   default : toute ressource est refusée par défaut, et l'accès doit être
│   explicitement accordé selon le rôle.
│
│   Côté frontend, le composant ProtectedRoute redirige vers une page
│   /unauthorized si l'utilisateur n'a pas le bon rôle.
│
│   On a 4 rôles : Admin avec accès complet, Sous-Admin sans droit de
│   suppression, Comptable limité aux paiements et factures, et Étudiant
│   qui ne voit que ses propres données."
│
└──────────────────────────────────────────────────────────────────────────────

┌─ À ÉCRIRE DANS LE RAPPORT ───────────────────────────────────────────────────
│
│  Le contrôle d'accès basé sur les rôles (RBAC) est appliqué en double
│  couche : les Firestore Security Rules implémentent un principe de moindre
│  privilège avec deny by default côté base de données, tandis que le
│  composant ProtectedRoute gère les restrictions d'interface côté client.
│  Quatre rôles sont définis avec des périmètres d'accès distincts : Admin,
│  Sous-Admin, Comptable et Étudiant. Les 21 accès refusés enregistrés
│  lors de la démonstration confirment l'efficacité du système face aux
│  tentatives d'accès non autorisés.

══════════════════════════════════════════════════════════════════════════════
  RÉPONSES AUX QUESTIONS FRÉQUENTES DU JURY
══════════════════════════════════════════════════════════════════════════════

Q : "Pourquoi le score est 35/100 et pas 100 ?"
R : "C'est normal. Le score est calculé sur les 24 dernières heures.
     Pendant la démo, nous avons volontairement simulé des attaques pour
     montrer que le système les détecte. Plus il y a d'attaques détectées,
     plus le score baisse — c'est le comportement attendu en mode alerte.
     En production normale, le score est entre 85 et 100."

Q : "Comment vous garantissez que les logs ne peuvent pas être falsifiés ?"
R : "Les règles Firestore sont configurées avec allow update: if false et
     allow delete: if false. Même un administrateur ne peut pas modifier
     ou supprimer un log existant. De plus, l'horodatage est généré
     par serverTimestamp() côté serveur, donc le client ne peut pas
     falsifier la date."

Q : "Est-ce que le WAF ralentit l'application ?"
R : "Non, le WAF est un middleware Express qui s'exécute en quelques
     microsecondes. Il analyse l'URL, les paramètres et le corps de la
     requête avec des expressions régulières optimisées. L'impact sur
     les performances est inférieur à 1 milliseconde par requête."

Q : "Quelle est la différence entre SIEM et WAF sur le dashboard ?"
R : "Le WAF est le pare-feu — il bloque les requêtes malveillantes en temps
     réel avant qu'elles touchent la base de données. Le SIEM, c'est le
     journal centralisé — il agrège tous les événements de sécurité, pas
     seulement le WAF, mais aussi les authentifications, les accès RBAC
     et les opérations RGPD, pour donner une vue d'ensemble à l'administrateur."
